# ICU检测报告处理系统 代码审查意见

**审查时间**: 2026-04-13 11:37:01
**审查范围**: src/ 目录下的重构版本代码
**版本**: v2.0.0

---

## 项目架构评估

重构后的系统采用了**配置文件驱动**的架构，比之前的硬编码版本有显著改进。核心结构：

```
src/
├── icu_report_processor/
│   ├── config/hospital_config.yaml  ← 配置文件（医院特定规则）
│   ├── classifiers/                  ← 分类器
│   ├── parsers/                      ← 解析器
│   ├── exporters/                    ← 导出器
│   └── processor.py                  ← 主处理器
└── run.py                            ← 入口
```

---

## 发现的问题

### 严重问题 (3)

#### 1. 去重功能直接删除文件，无确认机制
**位置**: `src/icu_report_processor/processor.py:146-152`

```python
for dup_file in files[1:]:
    try:
        dup_file.unlink()  # ← 直接删除，没有用户确认！
        duplicates_count += 1
```

**风险**: 如果误判或 MD5 碰撞，会导致数据永久丢失。

**建议**: 增加 `--dry-run` 模式或用户确认后再执行删除。

---

#### 2. 表格数据提取正则过于严格
**位置**: `src/icu_report_processor/config/hospital_config.yaml:322`

```yaml
row_pattern: '\d+\s+\*?([^\d]+?)\s+([\d<>.]+)\s*([\u2191\u2193]?)\s+([^\s]+)'
```

**问题**: 假设每行必须以数字序号开头。如果不同医院的 PDF 格式是：
- `项目名称  结果  参考区间` （无序号）
- `1. 项目名  结果  参考区间`  （点号分隔）

就会匹配失败。

**建议**: 支持多种 row_pattern 或使用更灵活的正则。

---

#### 3. 字段映射存在硬编码
**位置**: `src/icu_report_processor/parsers/base.py:134-144`

```python
field_map = {
    'patient_name': 'patient_name',   # ← 硬编码映射
    'patient_id': 'patient_id',
    ...
}
```

**问题**: 配置文件中定义的 `field_mapping`（中文字段名）实际上没有被使用，代码中有硬编码的英文字段映射。如果要支持多家医院，需要改代码。

**建议**: 使用配置文件中的 field_mapping 动态映射字段。

---

### 中等问题 (5)

#### 4. 分类器关键词重叠可能误判
**位置**: `src/icu_report_processor/classifiers/base.py:103-128`

```python
# 关键词匹配（每个匹配 +10 分）
keywords = config.get('classification_keywords', [])
for keyword in keywords:
    if keyword in text:
        score += 10
```

**问题**: 如果一份报告同时包含多个报告类型的特征词，会按分数最高者分类，可能不准确。

**示例**:
- 心脏超声报告可能同时提到 BNP
- 炎症因子报告可能包含生化指标

**建议**: 增加互斥关键词机制，或提高特征指标的区分权重。

---

#### 5. 参考区间模式不完整
**位置**: `src/icu_report_processor/config/hospital_config.yaml:324-329`

```yaml
reference_patterns:
  - '^[\d.]+-[\d.]+$'           # 7.35-7.45
  - '^[\d.]+~[\d.]+$'           # 7.35~7.45
  - '^[<>]=?[\d.]+$'            # <50, >=10
```

**缺失的模式**:
- `≤` 和 `≥` 符号（如 `≤10`）
- 文字描述（如 `正常`, `阴性`）
- 范围格式如 `(0.0-10.0)`

**建议**: 补充更多参考区间格式。

---

#### 6. PDF 文本提取方法单一
**位置**: `src/icu_report_processor/pdf_utils.py:84-100`

```python
def extract_text_from_pdf(pdf_path: Path, method: str = 'pdfplumber'):
    if method == 'pdfplumber':
        return extract_text_with_pdfplumber(pdf_path)
    # pdftotext 定义了但从未被调用
```

**问题**: `extract_text_with_pdftotext` 定义了但 `method='pdfplumber'` 是默认值，导致 pdftotext 永不执行。部分扫描版 PDF 可能两个都失败。

**建议**: 增加自动降级机制，pdfplumber 失败后尝试 pdftotext。

---

#### 7. 时间戳提取只取第一个
**位置**: `src/icu_report_processor/pdf_utils.py:172-187`

```python
matches = re.findall(pattern, text)
if matches:
    return matches[0]  # ← 只取第一个
```

**问题**: 如果 PDF 中有多个时间（采集、接收、报告），只取第一个可能不是期望的那个。

**建议**: 返回所有匹配的时间，由调用方根据优先级选择。

---

#### 8. 没有处理 PDF 加密或损坏
**位置**: `src/icu_report_processor/pdf_utils.py` 和 `processor.py`

**问题**: 加密的 PDF 或损坏的文件会导致提取失败。代码只是打印错误然后跳过，没有：
- 重试机制
- 备用提取方法
- 错误日志记录

**建议**: 增加错误处理和日志记录机制。

---

### 轻微问题 (2)

#### 9. 冗余代码
`extract_reports.py` 和 `rename_pdfs.py` 等旧版本文件仍然存在于项目根目录，可能造成混淆。

---

#### 10. Config 单例模式无法热更新
**位置**: `src/icu_report_processor/config/__init__.py:174-191`

```python
_config_instance: Optional[Config] = None

def get_config(config_path: Optional[str] = None) -> Config:
    global _config_instance
    if _config_instance is None or config_path is not None:
        _config_instance = Config(config_path)
    return _config_instance
```

**问题**: 如果配置路径为 `None` 且实例已创建，调用 `get_config()` 会忽略新加载的配置。

## 核心风险

1. **去重删除文件不可逆** — 需要增加确认机制
2. **正则表达式灵活性不足** — 换一家医院可能完全无法提取数据
3. **字段映射硬编码** — 要支持多家医院需要改代码

---

## 结论

这套系统**在当前医院的固定 PDF 格式下可以使用**，但距离真正的"通用"还有较大差距。需要将更多规则外化到配置文件中，而不是硬编码在 Python 代码里。

---

## 改进建议优先级

| 优先级 | 问题 | 建议 |
|--------|------|------|
| P0 | 去重删除风险 | 增加 dry-run 模式或确认机制 |
| P0 | 字段映射硬编码 | 动态使用配置文件中的 field_mapping |
| P1 | 正则表达式灵活性 | 支持多种 row_pattern 或更灵活的正则 |
| P1 | PDF 提取失败处理 | 增加自动降级和错误日志 |
| P2 | 参考区间模式 | 补充更多格式 |
| P2 | 分类器误判 | 增加互斥关键词机制 |
| P3 | 清理冗余代码 | 删除旧版本文件 |
| P3 | 配置热更新 | 改进单例模式 |
